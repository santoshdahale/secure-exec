# Wasmer Runtime Architecture: JS to Rust Glue

## Overview

The wasmer-js project bridges JavaScript to the Wasmer WebAssembly runtime through **wasm-bindgen**. The Rust code is compiled to WebAssembly using `wasm-pack`, and JavaScript can call into it directly.

```
┌───────────────────────────────────────────────────────────────────┐
│  JavaScript (Browser/Node.js)                                     │
│  Wasmer.fromFile(bytes) → Promise<Wasmer>                        │
└───────────────────────────────┬───────────────────────────────────┘
                                │ wasm-bindgen
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│  wasmer-js (Rust → WASM)                                         │
│  #[wasm_bindgen] js_from_file() → BinaryPackage → Wasmer struct  │
└───────────────────────────────┬───────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│  wasmer-wasix (Rust crate)                                       │
│  WasiRunner, WasiEnv, BinaryPackage, spawn_exec()                │
└───────────────────────────────┬───────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────┐
│  wasmer (Rust crate)                                             │
│  Engine, Module, Store - actual WASM compilation & execution     │
└───────────────────────────────────────────────────────────────────┘
```

## 2. Rust wasm-bindgen Layer

**File: `src/wasmer.rs:87-100`**

```rust
#[wasm_bindgen(js_name = "fromFile")]
pub async fn js_from_file(
    binary: Uint8Array,
    runtime: Option<OptionalRuntime>,
) -> Result<Wasmer, Error> {
    let bytes = binary.to_vec();
    if bytes.starts_with(b"\0asm") {
        // Raw WASM file - wrap in synthetic package
        Wasmer::from_wasm(bytes, runtime)
    } else {
        // WEBC package file - load container
        Wasmer::from_file(bytes, runtime).await
    }
}
```

**Key annotations:**
- `#[wasm_bindgen(js_name = "fromFile")]` - Maps Rust function to `Wasmer.fromFile` in JS
- `Uint8Array` - Automatically converted from JS `Uint8Array` to Rust vector
- `async fn` - Returns a JavaScript `Promise`

## 3. Loading a WEBC Package

**File: `src/wasmer.rs:126-134`**

```rust
async fn from_file(binary: Vec<u8>, runtime: Option<OptionalRuntime>) -> Result<Self, Error> {
    let runtime = runtime.unwrap_or_default().resolve()?.into_inner();

    // Detect WEBC format version
    let version = webc::detect(&mut (&*binary))?;

    // Parse WEBC container (zstd-compressed TAR with manifest + atoms)
    let container = webc::Container::from_bytes_and_version(binary.into(), version)?;

    // Convert to BinaryPackage (resolves dependencies, extracts commands)
    let pkg = BinaryPackage::from_webc(&container, &*runtime).await?;

    Wasmer::from_package(pkg, runtime)
}
```

## 4. Creating the Wasmer Struct with Commands

**File: `src/wasmer.rs:136-161`**

```rust
fn from_package(pkg: BinaryPackage, runtime: Arc<Runtime>) -> Result<Self, Error> {
    let pkg = Arc::new(pkg);
    let commands = Commands::default();  // JavaScript Object

    // For each command in the package, create a Command wrapper
    for cmd in &pkg.commands {
        let name = JsString::from(cmd.name());
        let value = JsValue::from(Command {
            name: name.clone(),
            runtime: Arc::clone(&runtime),
            pkg: Arc::clone(&pkg),
        });
        // Expose to JS: wasmer.commands.commandName
        Reflect::set(&commands, &name, &value).map_err(Error::js)?;
    }

    let entrypoint = pkg.entrypoint_cmd.as_deref().map(|name| Command {
        name: name.into(),
        pkg: Arc::clone(&pkg),
        runtime,
    });

    Ok(Wasmer { entrypoint, commands, pkg: None })
}
```

## 5. Running a Command

**File: `src/wasmer.rs:199-267`**

When you call `command.run()` in JavaScript:

```rust
#[wasm_bindgen]
impl Command {
    pub async fn run(&self, options: Option<SpawnOptions>) -> Result<Instance, Error> {
        // 1. Create pipes for stdin/stdout/stderr
        let (stdin, stdin_tx) = options.stdin.then(input_pipe).unzip();
        let (stdout, stdout_rx) = output_stream();
        let (stderr, stderr_rx) = output_stream();

        // 2. Configure WasiRunner
        let mut runner = WasiRunner::new();
        runner
            .with_args(args)
            .with_envs(env.iter().cloned())
            .with_stdin(Box::new(stdin_tx.unwrap_or_else(|| Box::new(Pipe::channel().0))))
            .with_stdout(Box::new(stdout_rx))
            .with_stderr(Box::new(stderr_rx));

        // 3. Spawn on dedicated thread
        let (exit_tx, exit_rx) = tokio::sync::oneshot::channel();
        tasks.task_dedicated(Box::new(move || {
            let result = runner.run_command(
                &command_name,
                &pkg,
                RuntimeOrEngine::Runtime(runtime)
            );
            let _ = exit_tx.send(ExitCondition::from_result(result));
        }))?;

        // 4. Return Instance with streams
        Ok(Instance { stdin, stdout, stderr, exit: exit_rx.shared() })
    }
}
```

## 6. Into wasmer-wasix: WasiRunner

**File: `../wasmer/lib/wasix/src/runners/wasi.rs`**

```rust
impl WasiRunner {
    pub fn run_command(
        &mut self,
        command_name: &str,
        pkg: &BinaryPackage,
        runtime_or_engine: RuntimeOrEngine,
    ) -> Result<(), Error> {
        // 1. Build WasiEnv from configuration
        let builder = self.prepare_webc_env(command_name, &wasi, pkg, runtime_or_engine, None)?;
        let env = builder.build()?;

        // 2. Load the WASM module for this command
        let cmd = package_command_by_name(pkg, command_name)?;
        let module = runtime.load_command_module(cmd).await?;

        // 3. Spawn execution
        let task_handle = spawn_exec_module(module, env, &runtime)?;

        // 4. Wait for completion
        let exit_code = task_handle.wait_finished().await?;
    }
}
```

## 7. Into wasmer-wasix: Module Loading

**File: `../wasmer/lib/wasix/src/runtime/mod.rs`**

```rust
pub trait Runtime {
    fn load_command_module(&self, cmd: &BinaryPackageCommand)
        -> BoxFuture<Result<Module, SpawnError>>
    {
        // Get WASM bytes + hash from command
        let data = HashedModuleData::from_command(cmd);

        // Try cache first, compile if miss
        load_module(self.engine(), self.module_cache(), &data)
    }
}

pub async fn load_module(
    engine: &Engine,
    module_cache: &dyn ModuleCache,
    module: &HashedModuleData,
) -> Result<Module, SpawnError> {
    let hash = module.hash();

    // Check cache
    if let Ok(cached) = module_cache.load(hash, engine).await {
        return Ok(cached);
    }

    // Compile from WASM bytes
    let compiled = Module::new(engine, module.wasm())?;

    // Save to cache
    module_cache.save(hash, engine, &compiled).await.ok();

    Ok(compiled)
}
```

```rust
impl Module {
    pub fn new(engine: &impl AsEngineRef, binary: &[u8]) -> Result<Self, CompileError> {
        Self::validate(engine, binary)?;
        unsafe { Self::from_binary_unchecked(engine, binary) }
```

## Summary

The architecture cleanly separates concerns:

1. **wasmer-js** - Browser-specific: Web Workers, Web Streams, HTTP fetching
2. **wasmer-wasix** - WASI/WASIX runtime abstraction, environment management
3. **wasmer** - Core WASM compilation and execution

The glue is primarily `#[wasm_bindgen]` annotations that automatically generate JavaScript bindings, combined with manual stream bridging (`src/streams.rs`) and a custom thread pool implementation (`src/tasks/`) for Web Worker-based parallelism.

```rust
pub(crate) enum GlobalScope {
    Window(Window),                    // Browser main thread
    Worker(WorkerGlobalScope),         // Web Worker context
    Other(js_sys::Object),             // Unknown (Node.js, Deno, etc.)

impl GlobalScope {
    pub fn current() -> Self {
        let global_scope = js_sys::global();
        match global_scope.dyn_into() {
            Ok(window) => GlobalScope::Window(window),
            Err(global_scope) => match global_scope.dyn_into() {
                Ok(worker_global_scope) => GlobalScope::Worker(worker_global_scope),
                Err(other) => GlobalScope::Other(other),  // Fallback for Node.js

        }

    pub fn hardware_concurrency(&self) -> Option<NonZeroUsize> { ... }
```

## WASM Execution: Native WebAssembly APIs

**Both browser and Node.js use the same mechanism: native `WebAssembly` APIs.**

The `wasmer` crate has a **JS backend** (`lib/api/src/backend/js/`) that wraps the native `WebAssembly` JavaScript APIs. There is **no Wasmer VM running in WASM** - it delegates entirely to the host's WebAssembly implementation.

### Compilation (Module Creation)

**File: `wasmer/lib/api/src/backend/js/entities/module.rs:70-83`**

```rust
pub(crate) unsafe fn from_binary_unchecked(
    _engine: &impl AsEngineRef,
    binary: &[u8],
) -> Result<Self, CompileError> {
    let js_bytes = unsafe { Uint8Array::view(binary) };

    // Uses native WebAssembly.Module constructor
    let module = WebAssembly::Module::new(&js_bytes.into()).map_err(...)?;

    Ok(unsafe { Self::from_js_module(module, binary) })
}
```

This calls the **native** `new WebAssembly.Module(bytes)` - the browser/Node.js compiles the WASM to machine code.

### What Actually Runs the WASM

| Environment | WASM Engine | How It Works |
|-------------|-------------|--------------|
| **Chrome** | V8 | V8's TurboFan JIT compiles WASM to x64/ARM machine code |
| **Firefox** | SpiderMonkey | IonMonkey/Cranelift compiles WASM to native code |
| **Safari** | JavaScriptCore | B3 JIT compiles WASM to native code |
| **Node.js** | V8 | Same as Chrome - V8 compiles WASM to native code |

### The wasmer Crate's Role

The `wasmer` crate in JS mode is essentially a **thin wrapper** around native WebAssembly APIs:

```
┌─────────────────────────────────────────────────────────┐
│  wasmer-js (Rust → WASM)                                │
│  Provides: WasiRunner, BinaryPackage, Runtime trait     │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐

│  Wraps: js_sys::WebAssembly::Module                     │
│         js_sys::WebAssembly::Instance                   │
│         js_sys::WebAssembly::Memory                     │
│         js_sys::WebAssembly::Table                      │
└─────────────────────┬───────────────────────────────────┘

│  JIT-compiled by the JS engine                          │
```

### Key Point: No Interpreter

There's **no WASM interpreter** running. The wasmer-js SDK:

1. **Doesn't compile WASM itself** - delegates to `WebAssembly.Module`
2. **Doesn't execute WASM itself** - delegates to `WebAssembly.Instance`
3. **Does provide WASI/WASIX** - implements system calls (filesystem, networking, etc.)

The actual WASM bytecode is compiled to native machine code by V8/SpiderMonkey/JSC, running at near-native speed.

### Contrast with Native wasmer

On native platforms (not in browser), the `wasmer` crate has different backends:

| Backend | Compiler | Description |
|---------|----------|-------------|
| `sys` + Cranelift | Cranelift | Fast compile, good runtime |
| `sys` + LLVM | LLVM | Slow compile, best runtime |
| `sys` + Singlepass | Singlepass | Fastest compile, baseline perf |
| **`js`** | **Browser's JIT** | **Delegates to native WebAssembly** |

The `js` backend is used when `wasmer` is compiled to `wasm32-unknown-unknown` target.
